## Что это за проект

**Dikanish** — система управления пищевым производством. Не CRM, не ERP в корпоративном смысле. Это инструмент микроменеджмента для конкретного завода: пельмени, котлеты, вареники, блинчики, манты, хинкали, чебуреки, голубцы.

**Микроменеджмент — это суть продукта.** Владелец хочет знать:
- кто сколько сделал сегодня
- где сейчас этот заказ
- сколько осталось сырья
- кто когда пришёл
- кому сколько заплатить

Не "общая аналитика". А конкретные цифры по конкретным людям и заказам прямо сейчас.

## Архитектура — как устроен код

### Всё состояние живёт в `App()`
Один корневой компонент держит все данные и передаёт через `AppContext`. Не Redux, не Zustand — просто useState + createContext. Это сделано намеренно: один файл, всё видно.

### Два типа хуков для данных

**`usePersisted(key, init)`** — данные на сервере, синхронизируются между всеми устройствами:
- При загрузке: GET `/api/state/:key`
- При изменении: POST `/api/state/:key`
- Polling каждые 6 секунд через `/api/updates` — смотрит что изменилось

**`useLocalStorage(key, init)`** — данные только в этом браузере (камеры)

### Навигация
`page` — строка. `renderPage()` — switch по имени. Добавить страницу = добавить `case` в switch и item в `navGroups`.

### Новое состояние
Добавляется в `App()` и в объект `ctx` в `useMemo`.

### Board mode
```js
if(new URLSearchParams(window.location.search).get("board")==="1"){
  return <OrdersBoardStandalone/>;
}
```
Стоит ПОСЛЕ всех хуков, перед `if(!currentUser)`. React Rules of Hooks — нельзя ставить раньше.

```js
JOB_TITLES = ["лепщица","фасовщица","грузчик","курьер","торговый представитель","техничка","завхоз","менеджер цеха","другое"]
```

## Новые сущности

**Батчи (dk_batches):** `{id, productId, quantity, producedAt, expiresAt, status, note, createdBy}`
Создаются автоматически при выпуске (applyOutput). Срок годности: 7 дней от даты выпуска.
Статусы: `["активна","отгружена","списана","забрали сотрудники"]`

**Дефекты/брак (dk_defects):** `{id, employeeId, productId, batchId, quantity, reason, date, comment, createdBy, createdAt}`
Причины: `["нарушение рецептуры","брак материала","производственная ошибка","брак упаковки","другое"]`

**Расчёт оплаты (dk_payroll):** `{id, employeeId, weekStart, basePay, piecePay, total, status, comment, createdBy, createdAt}`
Статусы: `["начислено","подтверждено к выплате","удержано","перенесено","причина подтверждена"]`
Расчёт недельной суммы — по выработке и посещаемости за период.

**Долги магазинов (dk_debts):** `{id, storeId, amount, remaining, description, date, dueDate, status, comment, payments:[...], createdAt}`
Статусы: `["активен","частично погашен","погашен"]`. Только admin/owner могут создавать/платить.

## Известные баги и что уже исправлено

### Исправлено
- `/api/updates` всегда возвращал 0 результатов — JS `Date.now()` в мс, SQLite `unixepoch()` в секундах. Фикс: нормализация в server.js (`sinceRaw > 1e10 ? Math.floor(sinceRaw/1000) : sinceRaw`)
- Board mode нарушал Rules of Hooks — стоял ДО хуков. Перенесён после всех `useState`/`usePersisted`
- `window.innerWidth` не реагировал на resize — заменён на `isMobile` state + resize listener

### Технический долг (не баги, но важно)
- App.jsx ~5000 строк в одном файле — работает, но тяжело навигировать
- Камеры в localStorage — не синхронизируются между устройствами (сделано намеренно пока камеры не используются)
- Нет HTTPS (нет домена)
- Нет резервных копий БД
- Нет печати/экспорта отчётов

## Фокус — микроменеджмент

Каждая фича должна давать владельцу **больше видимости** в реальном времени:

- Кто сейчас работает → задания с временными метками
- Сколько сделано за смену → выпуск по рабочим
- Где завис заказ → история статусов с именами
- Сколько осталось сырья → предупреждения до того как кончится
- Кто опоздал → отметки посещаемости

**Плохая фича**: добавляет кнопку которую нажимают раз в месяц.
**Хорошая фича**: показывает что-то важное без того чтобы специально искать.

## Психология заказчика

**Тип**: холерик с прагматичным фильтром. Решает через прецеденты, не через логику.

**Убеждается только делом**: живая фича в продакшне > любой аргумент словами.

**Ключевые триггеры**:
- Контроль: каждая фича должна давать ему больше видимости
- Время: не тратить его на объяснения — показывать конкретику
- Доверие: он уже обжигался на разработчиках — счётчик невыполненных обещаний есть

**Говорить его языком**: "заказ", "отгрузка", "смена", "рабочий". Не "эндпоинт", не "компонент".

## Как работать с кодом

### Добавить новую страницу
1. Написать `const MyPage = () => { ... }` перед `App()`
2. Добавить `case "myPage": return <MyPage/>;` в `renderPage()`
3. Добавить `{id:"myPage", label:"Название", ok:true}` в нужную группу `navGroups`

### Добавить новое состояние
1. В `App()`: `const [myData, setMyData] = usePersisted("dk_my_key", INIT_MY_DATA);`
2. В `ctx` useMemo: добавить `myData, setMyData`
3. В деструктуризации нужного компонента: `const {myData, setMyData} = useContext(AppContext);`

### Правило Rules of Hooks
Все `useState`, `useEffect`, `usePersisted`, `useCallback`, `useMemo` — только в начале функции, до любых условных return. Нарушение = белый экран + ошибка в консоли.
